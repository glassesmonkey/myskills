# web-access Skill (OpenClaw edition)

## 核心原则

**联网任务不是“有没有工具”的问题，而是“有没有正确切换工具”的问题。**

这个 skill 的职责，不是替 Agent 规定固定步骤，而是校准 Agent 在联网场景中的思考方式，并补齐最小完备能力集。

> Skill = Agent 策略哲学 + 最小完备工具集 + 必要事实说明

---

## 浏览哲学

### ① 先定义成功标准

拿到任务后，先回答：

- 任务到底要拿到什么？
- 是要**发现信息源**、**读取内容**、还是**执行网页操作**？
- 什么结果才算完成？

后续所有切换路径，都以这个成功标准为锚点。

### ② 选一个最可能直达的起点

不是所有联网任务都该从搜索开始。

- 已知 URL 且目标是读正文 → 先走可读提取
- 已知是登录态 / 动态站 / 强交互站 → 直接进浏览器
- 只是要找信息源 → 先搜索

### ③ 用结果校验路径，而不是死磕工具

每一步结果都是证据，不只是成功/失败二元信号。

- 搜索没命中，可能不是关键词错，而是目标本身不存在或搜索不是对的入口
- fetch 返回空，不一定是页面没内容，也可能是 JS / 反爬 / 阅读器提取错区块
- 平台显示“内容不存在”，不一定真不存在，也可能是访问方式错了
- 已经拿到 DOM 文本时，不必为“像人点开”而额外折腾

### ④ 达成即停，不追求表演式完整

满足成功标准就收手，不做无意义补操作。

## 最小完备工具集：搜 / 读 / 做

### A. 搜：发现信息源

适用场景：
- 不知道目标 URL
- 需要找官网、原始出处、相关新闻、文档入口

优先能力：
- 已安装 `web-discovery` skill（需要搜索发现时优先）
- 必要时浏览器内站内搜索

原则：
- **搜索是发现入口，不是事实证明。**
- 涉及核实任务时，搜索只负责帮你找到一手来源。

### B. 读：读取公开内容

适用场景：
- 已知 URL
- 目标是文章、博客、文档、帮助中心、公开网页正文

优先顺序：
1. `web_fetch`
2. `scripts/read_url.py`（Jina → Scrapling fallback）
3. `pdf`（如果目标是 PDF）
4. 浏览器（如果前几种都不可靠）

#### `scripts/read_url.py` 路由原则

- **先试 Jina Reader**：最省 token，通常最干净
- **Jina 失败再试 Scrapling + html2text**
- 只返回正文，不返回整页 HTML 垃圾

Jina 失败条件：
- 请求失败 / 超时
- 内容太短
- 返回错误页、限流页、验证码页
- 明显抽错区块

#### 读页面时的事实说明

- Jina 非常适合文章、博客、文档、公告、帮助页
- 对商品页、仪表盘、强交互页面，Jina 经常抽错内容
- `web_fetch` 适合轻量读取，但不保证总能拿到最优正文
- 公开页面如果被轻度反爬，直接抓原始 HTML 再抽正文可能更稳

### C. 做：在网页上执行操作

适用场景：
- 登录态
- 动态渲染
- 点击、输入、选择、上传、滚动
- 页面内容必须经交互后才出现

优先能力：
- OpenClaw `browser` 工具

原则：
- 不要把浏览器当成最后兜底；对某些站，它才是一开始就该选的正路
- 先 `snapshot` 看结构，再 `act`
- 在同一个 tab 内保留 `targetId`，减少上下文漂移
- 需要登录态时，按真实需求选 profile：
  - **默认隔离浏览器**：适合公开站点
  - **`profile=user`**：已有登录态重要，且用户在场可批准 attach
  - **`profile=chrome-relay`**：用户明确要 relay / extension attach-tab 流程时再用

详细模式见：`references/openclaw-browser.md`

## OpenClaw 路由表

| 场景 | 首选路径 |
|---|---|
| 已知公开 URL，读文章/文档正文 | `web_fetch` 或 `scripts/read_url.py` |
| 已知公开 PDF | `pdf` |
| 不知道 URL，只知道主题 | 搜索能力 / 站内搜索 |
| 需要登录态 | `browser` + 合适 profile |
| 页面要点击后才出内容 | `browser` |
| 页面主要是图片/视频 | `browser` 截图 / 抽媒体 + `image` |
| 多个独立目标 | 子 agent / isolated session 分治 |

## 程序化读取 vs GUI 交互

网页访问有两种思路：

### 程序化读取

例如：
- `web_fetch`
- Jina Reader
- 直接抓 HTML
- `pdf`

优点：
- 快
- 便宜
- 结构化
- 适合批量

缺点：
- 容易被反爬、抽错、拿不到动态内容

### GUI / 浏览器交互

例如：
- 打开页面
- 点击按钮
- 输入表单
- 滚动触发懒加载
- 截图理解页面

优点：
- 最接近真实用户路径
- 适合登录态、动态站、复杂交互

缺点：
- 慢
- 贵
- 上下文更重

**决策原则：能直接拿到目标，就别多走一层；明显需要交互时，也别在静态路径上浪费时间。**

## 核实类任务

核实不是“找更多二手报道”，而是找到一手来源。

| 信息类型 | 一手来源 |
|---|---|
| 官方政策 / 公告 | 发布机构官网 |
| 产品能力 | 官方文档 / 源码 / 产品页 |
| 学术结果 | 论文 / 官方机构 |
| 平台声明 | 官方账号 / 官方页面 |

若未找到官方原文，才退而求其次引用权威媒体，并明确告诉用户这属于转述。

## 子 Agent / 并行分治

当任务有多个**彼此独立**的目标时，鼓励分治。

适合分治：
- 多个竞品官网调研
- 多个候选链接可并行验证
- 多个平台独立信息收集

不适合分治：
- 强依赖前一步结果的链式任务
- 简单到几次读取就结束的任务

### Prompt 规则

给子 Agent 的 prompt 要写**目标**，少写**步骤**。

好：
- “获取这个品牌在小红书上的近期用户反馈，并总结主要观点。”

差：
- “先搜索小红书，再 fetch，再……”

因为动词本身会给子 Agent 锚定错误路径。比如你写“搜索”，它就可能执着于搜索引擎，而不是浏览器站内搜索。

## 站点经验沉淀

将已验证的站点经验按域名存到：

- `references/site-patterns/{domain}.md`

内容只写：
- 平台特征
- 有效模式
- 已知陷阱
- 发现日期

这些经验是**可能有效的先验**，不是永久真理。站点改版或风控变动后，允许回退到通用模式重新验证。

---

## 必要事实说明

- 站点生成的链接通常比你手工构造的 URL 更可靠，因为它自带隐式上下文
- 平台的“内容不存在”提示不一定可信，可能是访问方式问题
- 有些页面肉眼看不到的内容，其实已在 DOM 中
- 截图前滚动到相关区域，能触发懒加载并提高提取质量
- 页面主信息在图片/视频里时，不要只盯文本抽取
- 浏览器工具超时或不可用时，不要在同一路径无脑重试，应换成直接读取、Jina、HTML 抽取、或延后浏览器步骤
